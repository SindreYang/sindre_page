产品文档写作方法

发表于 2025-01-22 分类于 产品学 Changyan：

自动摘要: 1.定范围 文档标题：写的是什么 定位清晰，内容明确，如还在修订中，需要在标题后加上版本号 文档目的：为什么要写 用清晰简洁的语言描述文档的 ……..

1. 定范围

  • 文档标题：写的是什么
    • 定位清晰，内容明确，如还在修订中，需要在标题后加上版本号
  • 文档目的：为什么要写
    • 用清晰简洁的语言描述文档的用途
  • 文档归属：写给谁的
    • 通常属于文档撰写人获得组织
  • 创建时间/更新时间：什么时间写/修订的
    • 创建时间恒定不变，更新时间随文档更新而改变
  • 其他范围：补充说明
    • 例如一些文档保密级别

2. 选模板

产品工作文档需有三大模板（BRD，MRD，PRD）以及产品体验报告，用户调研报告，行业分析报告等

  • BRD—>What
    • 定义：商业需求文档，以”做什么“为核心
    • 服务对象：公司高层
    • 作用：描述商业闭环，获得高层支持，达成产品商业目标共识
  • MRD—>Why
    • 定义：市场需求文档，以”为谁做“为核心
    • 服务对象：市场和运营部门
    • 作用：描述市场，描述用户，为后续工作方向说明和工作指导
  • PRD—>How
    • 定义：产品需求文档，以：”怎么做“为核心
    • 服务对象：产品研发部
    • 作用：需求逻辑产品化，产品逻辑实体化：

BRD撰写–ppt

  1. 方案背景：发现什么需求，满足需求设计了什么方案，方案具有什么样的优势。
    1. 时间优势：我们最先发现需求并突出了可行的解决方案，可以率先打入市场从而占用市场。
    2. 技术优势：执行这个方案，我们具备哪些技术优势
    3. 经验优势：我们具备哪些经验优势
    4. 资源优势：我们拥有哪些经验优势
  2. 方案预测：分为方案阶段预测，未来价值预测
    1. 方案阶段预测：描述方案所经历的阶段，以及描述每个阶段所要达成的阶段性目标。
    2. 未来价值预测：描述整个方案在未来完成后可能产生的商业价值
  3. 产品规划：分为需求逻辑产品化闭环和产品架构图
    1. 需求逻辑产品化闭环：具体做出什么样的产品，这个产品可以是个系统或者一个APP
    2. 产品架构图：根据介绍需求，选择展示产品的业务架构图，功能架构图，信息架构图或者混合架构图
  4. 盈利模式：产品盈利闭环和盈利数据
    1. 产品盈利闭环：产品盈利方式有哪些，是否具备可持续性。
    2. 盈利数据：让盈利模式更有说服性
  5. 收益和成本：收益预估和成本预估
    1. 收益预估：需基于盈利模型中的盈利数据做更全面的描述，即这个产品有哪些盈利点，整个盈利模式可以产生多大的收益。
    2. 成本预估：需要罗列出各种成本支出，包括但不限于方案执行过程中的各种成本，以及后期产生的运营成本和维护成本，最后还需要的出中的成本支持，作为投入产出比的计算依据
  6. 风险与对策：风险种类和应对方法
    1. 风险种类：主要描述从方案立项到项目执行中所存在的风险，基于不同的方案背景，这些风险包括但不限于政策风险，行业风险，资本风险，经济风险，公司风险，市场风险，技术风险等。
    2. 应对方法：包括但不限于接受，规避，降低，转移，共担等。

MRD撰写–ppt

  1. 市场说明
    1. 市场规模：直接决定了产品最终的利润大小，核心模块，能直接用现金流的产品一般用交易额描述市场规模，间接产生现金流的产品则通过用户量来描述市场规模
    2. 市场增速：计算市场规模在时间维度上的变化，建议将市场规模和市场增速放在一起用图表进行展示
    3. 市场潜力：对市场需求所能达到的最大数值的测算，基于市场规模和增速，给出合理的市场潜力预测值
  2. 用户说明
    1. 用户定位：最好能用一句话清楚的描述，如淘宝用户定位是18-65岁有网购需求的消费人群
    2. 认知程度：新兴产品或行业需要调研分析的一个重要程度。说明了用户对新技术，新产品的态度以及预期，是否愿意接受和尝试。决定产品前期的设计决策，以及中后期的营销，运营决策
    3. 用户画像：主要介绍用户的人群分布特征，例如，性别分布，年龄分布，地区分布，职业分布，收入分布等。
    4. 需求场景：主要描述用户与产品交互过程中需求发生的主场景，以及要描述清楚在该场景下产品是如何满足用户需求的
    5. 总结：对整个模块与分析过程进行整理，形成完整的用户分析总结
  3. 产品说明
    1. 产品说明：说明产品的定位和产品的核心目标，应当简洁，形象准确的传达产品的核心和边界，例如微信小程序定位自身是一种链接用户与服务的工具，对外可以占领市场，提高用户对产品的认知度。
    2. 产品定位：以静态视角描述产品时什么，例如，微信的核心目标是连接人与人，满足人们的社交需求。
    3. 产品核心：以动态视角描述产品会怎样
    4. 产品架构：描述实体产品的设计框架，可以是产品业务架构，产品功能架构，产品混合架构等
    5. 产品路线：描述从需求到产品需要经历哪些阶段，应当有明确的计划，例如：需求输出，产品方案设计，产品上线发布，产品启动，产品运营等阶段，需说明每个阶段的目标及时间节点。
  4. 竞品说明：
    1. 竞争格局：主要描述当前市场同行业内存在竞争关系的所有产品以及其所属公司的信息。按某个具有代表性的指标进行排名，如用户量，市场份额，交易额，以不同的竞争维度进行分析，如产品获客的竞争，用户体验竞争，产品价格的竞争以及其他竞争等。总结出当前市场的竞争格局
    2. 代表性竞品分析：针对一个或者两个典型的竞品，分别分析这些竞品的产品定位和产品核心目标，主要分析产品定位，产品核心目标，差异性对比。
    3. 潜在替代品：主要分析当前市场上是否有自身产品的替代品或替代方案，如有，分析替代品的威胁程度，以及给出应对方案，最后结合竞品分析的内容和分析结果进行整理和总结

PRD撰写–word

  1. 基本规范：
    1. 文档封面：尽力只包含简洁的标题和版本号，文档注意添加页脚和页眉元素辅助说明内容
    2. 文档目录
    3. 更新记录
  2. 文档概述
    1. 背景概述：描述整个项目的初始需求背景，目的是让未参与过项目的人能通过背景概述了解该项目
    2. 范围和边界：定义了项目的影响规模，包括产品边界，业务边界，运营边界，技术和人力资源边界。
    3. 阅读对象：主要用来限制哪些组织或个人可以阅读此文档，有利控制文档的传播效率和查看权限。
    4. 术语定义：主要用来详细介绍文档中可能出现的专业术语，确保所有术语都有唯一的注解。
  3. 产品说明
    1. 产品简介：主要介绍产品的基本情况，包括产品定位，用户使用场景以及产品的功能介绍等。
    2. 产品流程：提供核心功能的业务流程图以及操作流程图，以方便阅读者快速了解产品的核心流程。
    3. 功能需求：核心，主要向研发人员说明产品的实现逻辑
      1. 功能简介：对功能进行简洁的描述
      2. 场景描述：用于描述功能的使用场景
      3. 功能优先级：明确功能得优先级
      4. 输入/前置条件：用于描述进行一个功能操作得前置条件，如收藏功能，需跟账户关联，所以前置条件就是用户已登录
      5. 需求描述：用于对原型和整个功能需求进行原型展示和详细得描述。
      6. 输出/后置条件：用于描述进行一个功能将产生得结果，例如：APP里面夜间模式，后置条件就是APP视觉风格变得适合夜间阅读
  4. 非功能需求
    1. 对功能需求得辅助和补充，非功能需求一般包括但不限于技术需求，运营需求，财务法务需求，其他需求等。
  5. 附录
    1. 指产品说明文档后于正文有关的其他资料和参考资料，不是必须的

3. 守规范

  • 命名规范
    • 通常命名规则为描述对象+版本号，如“xxx系统产品需求文档 V1.0.0“
    • 如果输出文档为最终版，则无需添加版本号
  • 表达规范
    • 避免使用口语化表达，如：”我认为”，我觉得”，“接下来我们”，“那咱们”，会降低文档的专业性和严谨性
  • 版本号规范
    • 版本更新内容包括：版本号，修订人，修订时间，修订内容，备注
    • 版本号采用“VX.Y.Z“的格式，X是主版本号，Y是次版本号，Z是修订号
    • 首次发布的版本命名为V1.0.0
      • 小规模修订时，为V1.0.0–>V1.0.1
      • 中等规模修订时，为V1.0.0–>V1.1.0
      • 大规模改动时，为V1.0.0–>V2.0.0
      • 如果前面数字叠加后，后面数字从归零计算，如V1.0.1 变为V1.1.0,则将V1.2.1更改为V2.0.0
